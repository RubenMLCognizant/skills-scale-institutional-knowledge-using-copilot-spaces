# Persona Onboarding Checklist

When a team member joins a project in any of the defined OctoAcme personas, use this checklist to ensure they have context, access, and clarity on their role.

## General Onboarding (All Personas)

- [ ] Share project vision, goals, and success metrics
- [ ] Review relevant process documentation (this guide and linked process docs)
- [ ] Clarify role responsibilities, expectations, and reporting relationships
- [ ] Grant access to project tools (GitHub, project board, shared documentation)
- [ ] Schedule intro meetings with key cross-functional collaborators
- [ ] Add to relevant Slack channels and communication lists
- [ ] Share decision logs and previous retrospective notes
- [ ] Review any applicable security, compliance, or brand guidelines

## Role-Specific Onboarding

### Developer
- [ ] Set up development environment and local repository
- [ ] Review Definition of Done and PR standards
- [ ] Pair with a senior engineer on first PR review
- [ ] Understand deployment pipeline and testing requirements

### Product Manager
- [ ] Review customer research, analytics, and metrics dashboards
- [ ] Understand prioritization framework and stakeholder constraints
- [ ] Schedule meetings with key stakeholders to establish communication cadence
- [ ] Review roadmap and previous product decisions

### Project Manager
- [ ] Review project plan, timeline, and key milestones
- [ ] Understand risk register and escalation paths
- [ ] Set up regular communication cadence (standups, syncs, status reports)
- [ ] Review stakeholder list and communication preferences

### UX/UI Designer
- [ ] Review design system, brand guidelines, and Figma workspace
- [ ] Understand product roadmap and user research findings
- [ ] Schedule design review kickoff with developers and product team
- [ ] Share current design artifacts and rationale

### DevOps / Platform Engineer
- [ ] Review infrastructure architecture and deployment pipeline
- [ ] Understand current monitoring, alerting, and on-call procedures
- [ ] Get access to infrastructure repos, CI/CD systems, and observability tools
- [ ] Review incident playbooks and rollback procedures

### Business Analyst
- [ ] Review current requirements documentation and backlog
- [ ] Understand stakeholder landscape and business drivers
- [ ] Schedule requirements gathering workshops
- [ ] Review acceptance criteria standards and Definition of Done

### Change Manager
- [ ] Identify all stakeholders who will be impacted by upcoming changes
- [ ] Review communication templates and previous change initiatives
- [ ] Schedule stakeholder engagement planning sessions
- [ ] Set up feedback collection mechanisms

### Support Lead
- [ ] Review common customer issues and support ticketing system
- [ ] Understand current product features and known limitations
- [ ] Schedule knowledge transfer from product and engineering teams
- [ ] Establish regular handoff cadence (weekly or pre/post release)
